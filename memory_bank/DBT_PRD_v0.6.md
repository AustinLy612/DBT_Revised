## 1. 产品背景与定位

本平台是一个面向研究场景的 DBT 技能教学与训练 Web 平台，目标用户为具有心理健康风险的中学生。平台通过大语言模型、RAG 知识库、图像生成工具与研究后台，为用户提供个性化、情景化、连续化的 DBT 技能教学。

平台当前优先支持 DBT 中的正念技能教学，后续需扩展至 DBT 四大核心技能模块：

1. 正念（Mindfulness）
2. 痛苦忍受（Distress Tolerance）
3. 人际关系效能（Interpersonal Effectiveness）
4. 情绪调节（Emotion Regulation）

本平台不是单纯的 AI 聊天机器人，而是一个包含用户信息采集、AI 个性化教学、RAG 文档检索、情景图片生成、自动测试、情绪记录、成就系统、风险识别、研究后台与数据导出的完整研究平台。

## 2. 产品目标

### 2.1 核心目标

1. 为具有心理健康风险的中学生提供结构化的 DBT 技能教学与训练。
2. 基于用户个人信息、当前状态、历史教学记录和情绪记录进行个性化教学。
3. 使用 RAG 知识库支持 DBT 技能教学与测试，避免 AI 仅依赖模型记忆生成内容。
4. 通过情景化教学和图像生成增强用户代入感。
5. 在教学完成后自动进入测试，不依赖用户主动选择。
6. 通过每题解析和允许重测机制提高学习效果。
7. 通过情绪记录功能追踪用户学习前后情绪变化。
8. 通过中文成就系统提升用户持续参与度。
9. 对高风险表达进行识别，并自动中断教学、记录事件、弹窗提示相关求助方式。
10. 支持研究后台查看和导出用户相关数据。
11. 作为研究平台，验证该教学链路是否能够帮助已筛入受试者产生实时或短周期情绪改善。

### 2.2 产品边界

1. 平台定位为研究用 DBT 技能教学与训练工具。
2. 平台不提供医学诊断。
3. 平台不替代心理咨询师、精神科医生、学校心理老师或紧急救助服务。
4. 平台在识别到高风险表达后，应停止普通教学流程，而不是继续进行技能教学。
5. 当前版本不实现知情同意页面或监护人同意页面；如研究流程需要相关材料，应在平台外部流程中处理。
6. 平台不承担受试者筛查职责；受试者由学校与精神卫生中心完成评估和筛入。
7. 平台默认运行于有实验人员陪同的研究环境中，高风险后的线下跟进由现场研究人员承担。

## 3. 目标用户

### 3.1 用户群体

目标用户为已由学校与精神卫生中心完成评估、并已筛入研究样本的中学生，覆盖年级范围为：

- 初一
- 初二
- 初三
- 高一
- 高二
- 高三

### 3.2 用户特征

该类用户可能具有以下特点：

1. 面临学业、人际、家庭或情绪方面的困扰。
2. 需要更低门槛、更自然、更具引导性的教学方式。
3. 对复杂、过度学术化或过长文本的接受度较低。
4. 需要在产品流程中嵌入明确的安全保护机制。
5. 适合通过情景化、例子化和选择题练习进行技能学习。
6. 使用场景以研究环境为主，通常在实验人员陪同下完成单次训练。

## 4. 用户角色

### 4.1 普通用户

普通用户为中学生使用者。其主要行为包括：

1. 注册和登录账号。
2. 完成注册后问卷。
3. 参与 AI DBT 技能教学。
4. 完成自动生成的情景选择题测试。
5. 进行情绪记录。
6. 查看情绪历史。
7. 查看已解锁成就。

### 4.2 AI 教学助手

AI 教学助手负责：

1. 收集本次教学所需信息。
2. 结合用户资料和历史记录选择 DBT 技能。
3. 调用 RAG 检索 DBT 技能文档。
4. 基于检索结果进行对话式教学。
5. 构建教学情景。
6. 调用图像生成工具生成教学情景图片。
7. 基于 RAG 检索结果和用户历史生成测试题。
8. 在测试题中调用图像生成工具生成题目情境图片。
9. 对用户每题答案进行即时解析。
10. 在测试失败后讲解错误原因并允许重测。
11. 在检测到高风险内容时停止教学流程。

### 4.3 报告查看用户

学校老师或其他研究者可作为“报告查看用户”使用前台只读报告能力。其权限边界如下：

1. 只能查看经授权范围内学生的个性化可视化报告。
2. 可以导出学生个性化报告。
3. 不可查看原始数据库。
4. 不可修改或删除原始研究数据。
5. 不可进入研究后台管理界面。

### 4.4 研究后台管理员

研究后台管理员可查看和导出平台数据，并可使用 MongoDB Compass 等管理工具进行研究数据管理。后台允许查看：

1. 用户注册信息。
2. 用户问卷信息。
3. 用户教学历史。
4. 用户聊天记录。
5. 用户测试记录。
6. 用户情绪记录。
7. 用户成就记录。
8. 用户风险事件记录。
9. RAG 文档状态和检索日志。

后台应支持将上述信息与对应使用者关联后导出。

研究后台管理员与报告查看用户的权限必须严格隔离：前者可接触原始数据与后台管理能力，后者仅可使用只读可视化报告页面和对应报告导出能力。

## 5. 核心产品流程

### 5.1 首次使用流程

1. 用户进入 Web 平台。
2. 用户输入研究邀请码并注册账号。
3. 用户设置自定义账号与密码。
4. 系统校验邀请码有效性并创建用户账号。
5. 系统引导用户填写注册后问卷。
6. 用户完成问卷后进入首页。
7. 用户可开始第一次 DBT 技能教学。

### 5.2 单次教学完整流程

单次教学流程必须自然推进，不应依赖用户主动选择是否进入测试。

默认使用节奏为每天 1 次、每次约 15-20 分钟，持续 2-3 周。系统默认自动推进教学和测试链路，但受试者可在任意阶段自主终止当前教学或测试。

#### Step 1：教学前情绪记录

用户开始教学前，前端弹窗提示用户记录当前情绪。用户通过 5 个 emoji 中的一个记录当前心情。

#### Step 2：信息收集

AI 通过自然对话收集用户本次状态和相关经历，并同时读取：

1. 注册问卷信息。
2. 历史教学记录。
3. 历史测试表现。
4. 历史情绪记录。
5. 历史聊天摘要或聊天记录。

#### Step 3：RAG 检索与个性化技能选择

AI 根据当前信息和历史记录，从当前支持的 DBT 正念完整技能集合中选择本次教学技能。选择前后均可调用 RAG 知识库检索相关技能文档，以确保教学内容基于已导入的 DBT 文档。

AI 应简要解释为什么选择该技能，但不能进行医学诊断或过度心理推断。

#### Step 4：对话式教学

AI 基于 RAG 检索到的技能文档进行教学，包括：

1. 技能名称。
2. 技能适用场景。
3. 技能核心概念。
4. 技能使用步骤。
5. 与用户情况相关的例子。
6. 简短练习。
7. 理解确认。

#### Step 5：教学情景与图片生成

AI 可根据教学内容和用户信息构建教学情景，并调用图像生成工具生成教学场景图片。

图片应符合以下要求：

1. 适合中学生。
2. 与教学技能相关。
3. 与用户信息和常见校园/家庭/人际场景相匹配。
4. 不包含血腥、惊悚、羞辱、暴力、自伤、自杀等刺激性内容。

#### Step 6：自动过渡到测试

教学完成后，AI 自动提示用户即将开始测试。系统随后自动跳转到测试页面并开始测试。

用户不需要点击“是否开始测试”或自行选择进入测试。

#### Step 7：测试

测试共 5 道题。题目形式为情景选择题。题目由 AI 根据用户信息、教学历史、本次教学内容和 RAG 检索结果自主设计。

测试过程中，AI 需调用图像生成工具，为题目生成符合情境的图片。

#### Step 8：逐题解析

用户每回答一题后，AI 应立即给出解析，包括：

1. 用户选择是否正确。
2. 正确答案。
3. 为什么该答案更合适。
4. 本题对应的 DBT 技能点。
5. 如果用户答错，应说明错误原因。

#### Step 9：测试结果

5 道题完成后，系统展示：

1. 正确题数。
2. 是否通过。
3. 每题结果回顾。
4. 本次学习掌握情况总结。
5. 是否触发成就。

#### Step 10：失败后的讲解与重测

若用户答对题数少于 4 题，则判定为未通过。未通过后：

1. AI 应总结主要错误原因。
2. AI 应重新讲解用户薄弱的技能点。
3. 系统允许用户重测，且不限制重测次数。
4. 重测题目应由 AI 重新生成，不应简单重复原题。

#### Step 11：教学后情绪记录

测试结束后，系统弹窗提示用户再次记录情绪。

#### Step 12：保存记录与更新成就

系统保存本次完整流程数据，包括教学、测试、情绪记录、成就变化、RAG 检索记录和风险标记等。

3. 问卷修改后，后续 AI 教学、测试生成和个性化推荐应使用最新问卷信息。
4. 历史教学记录不需要因问卷修改而回溯改写。

### 6.3 AI DBT 技能教学

#### 6.3.1 功能描述

AI 根据用户信息和历史记录进行个性化 DBT 技能教学。当前版本支持 DBT 正念板块下的全部技能，具体教学哪个技能由 AI 根据用户信息、教学历史、测试表现和情绪记录动态决定，不预设固定教学顺序。后续版本需扩展至 DBT 四大模块。

#### 6.3.2 当前正念技能范围

当前正念模块应覆盖 DBT 正念板块下的完整技能集合。默认包括：

1. 智慧心智（Wise Mind）
2. 观察（Observe）
3. 描述（Describe）
4. 参与（Participate）
5. 不评判地（Nonjudgmentally）
6. 一次只做一件事（One-mindfully）
7. 有效地行动（Effectively）

如果后续导入的 DBT 技能文档采用不同命名或拆分方式，则以 RAG 知识库中的正式教学文档为准。

#### 6.3.3 功能要求

| AI-001 | 教学前信息收集 | P0 | AI 通过对话收集本次教学所需信息 |
| AI-002 | 历史记录读取 | P0 | AI 可读取同账号历史教学和测试记录 |
| AI-003 | 个性化技能选择 | P0 | 当前从正念完整技能集合中选择，后续扩展至四模块 |
| AI-004 | RAG 检索 | P0 | 教学前和测试前检索 DBT 技能文档 |
| AI-005 | 对话式教学 | P0 | AI 以自然对话方式讲解技能 |
| AI-006 | 情景化教学 | P0 | AI 结合用户信息构建教学情景 |
| AI-007 | 自动过渡到测试 | P0 | 教学完成后自动进入测试链路 |

### 6.4 RAG 知识库与检索功能

#### 6.4.1 功能描述

AI 需支持 RAG（Retrieval-Augmented Generation）功能，用于 DBT 技能教学与测试所需文档的存储、检索和使用。AI 在生成教学内容、选择技能、设计情景题、生成答案解析时，应优先基于 RAG 知识库中的 DBT 技能文档，而不是仅依赖模型自身记忆。

#### 6.4.2 知识库内容

当前已有 DBT 技能文档，但仍需要整理和补充后再导入 RAG 知识库。整理目标是将原始 DBT 文档转化为适合检索、教学、测试生成和答案解析的结构化知识。

RAG 知识库应至少支持存储以下内容：

1. DBT 正念技能教学文档。
2. 后续扩展的痛苦忍受技能教学文档。
3. 后续扩展的人际关系效能技能教学文档。
4. 后续扩展的情绪调节技能教学文档。
5. 每个技能的适用场景、步骤、示例和注意事项。
6. 用于测试题生成的题目设计规范。
7. 用于中学生表达风格适配的教学规范。
8. 风险内容处理规范。

#### 6.4.3 RAG 使用场景

| 场景 | RAG 使用方式 |
|---|---|
| 技能选择 | 检索可用 DBT 技能、适用场景和先修关系 |
| 教学内容生成 | 检索对应技能的定义、步骤、示例和练习方式 |
| 情景生成 | 检索技能适用场景，结合用户资料生成安全情景 |
| 测试题生成 | 检索技能要点和常见误区，生成情景选择题 |
| 答案解析 | 检索正确使用方式和错误应用方式，用于逐题讲解 |
| 重测生成 | 检索用户薄弱技能点，生成不重复的新题 |
| 风险处理 | 检索风险提示和中断策略文档 |

#### 6.4.4 DBT 文档整理要求

导入 RAG 前，应对现有 DBT 技能文档进行整理和补充，至少包括：

1. 按 DBT 模块分类：正念、痛苦忍受、人际关系效能、情绪调节。
2. 按具体技能拆分：每个技能独立成文或独立片段集合。
3. 为每个技能补充适合中学生理解的简明解释。
4. 为每个技能补充适合中学生的典型应用场景。
5. 为每个技能补充正确使用方式与常见误区。
6. 为每个技能补充可用于测试题生成的情景题设计要点。
7. 为每个技能标注元数据，包括模块、技能名称、适用场景、难度、是否适合初次学习等。
8. 风险处理内容应作为独立规范文档存储，不与普通技能教学文档混用。

#### 6.4.5 RAG 功能要求

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---:|---|
| RAG-001 | 文档上传 | P0 | 后台可上传 DBT 技能文档 |
| RAG-002 | 文档切分 | P0 | 系统将文档切分为可检索片段 |
| RAG-003 | 向量化存储 | P0 | 文档片段进入向量数据库或等效检索系统 |
| RAG-004 | 关键词检索 | P0 | 支持基于技能名、模块名、关键词检索 |
| RAG-005 | 语义检索 | P0 | 支持基于用户问题和教学目标进行语义检索 |
| RAG-006 | 混合检索 | P1 | 支持关键词 + 语义的混合检索 |
| RAG-007 | 检索结果注入提示词 | P0 | 将相关文档片段提供给 AI 生成教学或测试内容 |
| RAG-008 | 文档版本管理 | P1 | 支持查看文档版本，避免使用过期内容 |
| RAG-009 | 检索日志保存 | P1 | 保存每次教学和测试使用的检索结果 |
| RAG-010 | 后台查看 RAG 文档 | P0 | 研究人员可查看已上传文档和状态 |

#### 6.4.6 RAG 约束

1. AI 教学和测试内容应优先依据 RAG 检索到的 DBT 文档生成。
2. 如果 RAG 检索结果不足，AI 应使用自身知识进行回答与教学，但禁止编造具体 DBT 技能细节。
3. 测试题的正确答案和解析必须能对应到 RAG 文档中的技能要点。
4. 图像生成 prompt 不应包含用户过度敏感信息。
5. 高风险处理文案应来自固定风险处理规范，而非完全由 AI 临时生成。

### 6.9 风险识别与危机处理

#### 6.9.1 功能描述

平台面向具有心理健康风险的中学生，因此必须设置高风险识别与中断机制。

#### 6.9.2 已确认规则

1. 风险识别由 AI 判断 + 关键词识别共同完成。
2. 不做风险分级。
3. 只有高风险情况需要自动停止教学。
4. 高风险触发后系统应记录事件。
5. 前端应弹窗进行风险提示。
6. 弹窗中显示中国大陆相关心理援助热线。
7. 弹窗中显示“请联系学校老师寻求帮助”等提示。
8. 不需要系统向研究人员发起额外通知；实验过程中默认有实验人员陪同。
9. 高风险中止仅作用于当前会话，不禁止受试者下次再次进入系统。

#### 6.9.3 高风险识别方式

系统采用双通道识别：

1. **关键词识别**：对用户输入进行关键词匹配，例如自伤、自杀、想死、活不下去、伤害自己、结束生命等相关表达。
2. **AI 语义识别**：AI 对用户表达进行语义判断，识别可能不直接包含关键词但表达出明确高风险意图的内容。

当关键词识别或 AI 语义识别判断为高风险时，触发高风险处理流程。

#### 6.9.4 高风险处理流程

1. 立即停止当前教学或测试流程。
2. 保存当前会话上下文。
3. 记录风险事件。
4. 前端弹出风险提示框。
5. 弹窗显示心理援助热线。
6. 弹窗提示用户联系学校老师、可信任的成年人或线下专业人员。
7. 不继续当前会话中的普通 DBT 教学。
8. 结束当前会话后，受试者后续仍可再次登录并开始新的会话。

#### 6.9.5 风险提示弹窗建议文案

> 你刚才表达的内容可能表示你正在经历比较危险或难以独自承受的状态。当前教学将先暂停。  
> 请尽快联系身边可信任的成年人，例如学校老师、班主任、学校心理老师、家长或监护人。  
> 你也可以拨打以下心理援助热线寻求帮助：  
> 全国统一心理援助热线：12356  
> 北京市心理援助热线：010-82951332；座机用户可拨打 800-810-1117  
> 如果你正处于立即危险中，请立刻联系身边成年人并寻求当地紧急帮助。

#### 6.9.6 风险事件记录字段

| 字段 | 说明 |
|---|---|
| risk_event_id | 风险事件 ID |
| user_id | 用户 ID |
| session_id | 教学会话 ID |

### 6.10 语音与文字交互

#### 6.10.1 功能描述

平台支持文字输入、语音输入和 AI 语音播报。用户语音输入后，系统将语音转写为文本，再进入 AI 教学流程。AI 回复默认以文字展示，同时支持语音播报。

当前不需要实时语音通话式交互。默认采用“录音/语音输入 → 语音转文字 → AI 回复 → 文字展示 + AI 语音播报”的异步链路。

#### 6.10.2 已确认规则

1. 支持用户语音输入。
2. 支持 AI 语音播报。
3. 系统保存语音转文字结果。
4. 系统不保存用户原始语音文件。
5. 教学、测试、风险识别和后台导出均基于语音转写文本，而不是原始音频。
6. 不需要实时语音通话式交互。

#### 6.10.3 模型与服务建议

| 能力 | 推荐方案 | 说明 |
|---|---|---|
| LLM 对话与教学生成 | MiniMax API，优先考虑 MiniMax-M2.7 或对应可用文本模型 | 用于教学对话、技能选择、题目生成、解析、总结和风险语义判断 |
| AI 语音播报 / TTS | MiniMax Speech 系列，优先考虑 speech-2.8-turbo 或同类低延迟模型 | 用于将 AI 回复转为语音播报 |
| 语音转文字 / ASR | 优先调研 MiniMax 是否提供适合本项目的 ASR 能力；如不满足，则使用中国大陆可用 ASR 服务作为替代 | 仅保存转写文本，不保存原始音频 |

说明：MiniMax 可作为 LLM 与语音播报的优先候选供应商。语音转文字部分需要在开发前进一步确认其 API 能力、稳定性、价格和中国大陆访问情况；如果 MiniMax ASR 不满足项目要求，应替换为中国大陆可用的专业 ASR 服务。

#### 6.10.4 功能要求

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---:|---|
| VOICE-001 | 文字输入 | P0 | 用户可通过文字进行对话 |
| VOICE-002 | 语音输入 | P0 | 用户可通过语音输入内容 |
| VOICE-003 | 语音转文字 | P0 | 语音内容转写后用于 AI 对话 |
| VOICE-004 | 转写文本保存 | P0 | 保存语音转写文本，不保存原始音频 |
| VOICE-005 | AI 语音播报 | P0 | AI 回复可通过 TTS 播报 |
| VOICE-006 | 语音播报开关 | P1 | 用户可选择开启或关闭 AI 语音播报 |
| VOICE-007 | 非实时语音链路 | P0 | 当前不支持实时语音通话，仅支持录音/转写/回复/播报 |

#### 6.10.5 待确认项

1. MiniMax 是否作为最终 LLM API 供应商。
2. MiniMax 是否作为最终 TTS 供应商。
3. 语音转文字是否最终采用 MiniMax；如不采用，需要选择替代 ASR 服务。

### 6.11 研究后台

#### 6.11.1 功能描述

研究后台用于研究团队管理员查看、管理和导出用户相关数据。

#### 6.11.2 已确认需求

后台允许查看全部研究相关数据，并可以与对应使用者的其他信息一同导出。研究数据为实名数据，仅管理员有权查看和导出原始数据。

#### 6.11.3 后台可查看内容

| 数据类型 | 是否可查看 | 是否可导出 |
|---|---:|---:|
| 用户账号信息 | 是 | 是 |
| 用户问卷信息 | 是 | 是 |
| 教学会话记录 | 是 | 是 |
| 聊天记录 | 是 | 是 |
| 教学摘要 | 是 | 是 |
| 测试题目 | 是 | 是 |
| 用户答案 | 是 | 是 |
| 每题解析 | 是 | 是 |
| 测试结果 | 是 | 是 |
| 重测次数 | 是 | 是 |
| 情绪记录 | 是 | 是 |
| 成就记录 | 是 | 是 |
| 风险事件记录 | 是 | 是 |
| RAG 文档与检索日志 | 是 | 是 |

#### 6.11.4 导出要求

导出数据应支持按用户聚合，即一个用户的数据可以与其问卷信息、教学历史、测试记录、情绪记录、成就记录、风险记录和 RAG 检索记录关联导出。

| Excel | 可选 | 适合研究人员人工查看和整理 |

#### 6.11.6 权限与日志

后台与报告能力涉及三类角色：

| 角色 | 权限 |
|---|---|
| 学生用户 | 只能访问自己的教学、测试、情绪和成就记录，不可查看可视化个体报告 |
| 报告查看用户（学校老师 / 其他研究者） | 只能在前台查看经授权学生范围内的个性化可视化报告，并可导出报告，不可修改或删除原始数据 |
| 管理员 | 可访问研究后台，查看原始用户数据、RAG 文档、风险记录，并执行原始数据导出、配置管理等操作 |

#### 6.11.7 导出与审计规则

1. 管理员可以执行数据导出。
2. 当前研究版本不要求匿名化或去标识化导出，实名研究数据仅限管理员导出。
3. 需要记录后台操作日志。
4. 操作日志至少记录：管理员 ID、操作类型、操作对象、导出范围、导出格式、操作时间。
5. 建议对敏感操作进行二次确认，例如批量导出和风险记录导出。

### 6.12 可视化报告与报告导出

#### 6.12.1 功能描述

平台除原始数据导出外，还应提供面向学校老师和其他研究者的前台可视化报告能力。该能力用于只读展示学生在研究过程中的关键指标，并支持导出个性化报告。

#### 6.12.2 报告内容范围

学生个性化可视化报告至少应包括：

1. 情绪记录时间线与教学前后情绪变化。
2. 技能学习次数与最近学习内容。
3. 测试通过情况、正确率与重测次数。
4. 已学习技能分布与阶段性掌握情况。
5. 成就解锁情况。
6. 风险事件摘要（如研究方案允许展示）。

#### 6.12.3 权限边界

1. 报告查看用户只能查看经管理员授权的学生范围。
2. 报告查看用户只能访问前台报告页，不可访问研究后台。
3. 报告查看用户可导出个体化报告，但不可导出原始 JSON / CSV / Excel 数据。
4. 学生本人当前版本不可查看个体化可视化报告。
5. 管理员既可导出原始数据，也可查看和导出可视化报告。

#### 6.12.4 功能要求

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---:|---|
| REPORT-001 | 前台可视化报告页 | P0 | 报告查看用户可在前台查看学生个性化可视化报告 |
| REPORT-002 | 情绪趋势展示 | P0 | 展示情绪记录时间线与前后变化 |
| REPORT-003 | 学习情况展示 | P0 | 展示技能学习次数、通过率、重测次数等 |
| REPORT-004 | 个性化报告导出 | P0 | 支持导出单个学生的个性化报告，优先 PDF |
| REPORT-005 | 报告权限控制 | P0 | 仅授权的报告查看用户和管理员可查看对应学生报告 |
| REPORT-006 | 报告访问日志 | P1 | 记录报告查看与导出行为，便于审计 |

## 8. AI / Agent 工作流设计

### 8.1 推荐架构

建议采用“后端状态机 + MiniMax LLM API（优先考虑）+ RAG + 工具调用”的混合架构。

原因：

1. 教学流程必须稳定、可控，适合由后端状态机管理。
2. 个性化教学、题目生成、解析和情景构建适合由 LLM 完成。
3. DBT 技能知识应来自 RAG 知识库，以提高内容一致性和可追溯性。
4. 当前已有 DBT 技能文档，但需要整理、补充、切分和结构化后再导入 RAG。
5. 图像生成、历史检索、风险识别、语音转文字、语音合成和数据保存适合工具化调用。
6. 面向中学生和心理健康风险场景时，应避免完全依赖 Agent 自主规划流程。

### 8.2 主要状态

1. pre_mood_recording
2. info_collection
3. skill_selection
4. rag_retrieval_for_teaching
5. teaching
6. teaching_image_generation
7. auto_transition_to_test
8. rag_retrieval_for_test
9. testing
10. per_question_explanation
11. test_summary
12. retest_review
13. post_mood_recording
14. completed
15. stopped_by_risk

### 8.3 工具调用能力

AI 可调用以下工具：

1. 用户问卷读取工具。
2. 历史教学记录检索工具。
3. 历史测试记录检索工具。
4. 情绪记录检索工具。
5. RAG 知识库检索工具。
6. DBT 技能库检索工具。
7. 图像生成工具。
8. 测试题生成工具。
9. 测试评分与解析工具。
10. 成就判定工具。
11. 风险关键词检测工具。
12. 风险事件记录工具。
13. 语音转文字工具。
14. 语音合成工具。
15. 后台数据导出工具。

## 10. 非功能需求

### 10.1 安全与隐私

1. 密码不得明文存储。
2. 普通用户只能访问自己的数据。
3. 后台访问需鉴权。
4. 权限分为学生、报告查看用户、管理员三类。
5. 管理员可以查看和导出原始研究数据。
6. 报告查看用户只能查看授权范围内的可视化报告并导出报告，不可修改或删除原始数据。
7. 不要求匿名化或去标识化导出。
8. 必须记录管理员后台操作日志，尤其是查看、导出、删除、修改配置等操作。
9. 用户数据包含敏感心理信息，应进行严格访问控制。
10. 后台导出默认支持 JSON，且应保留用户、教学、测试、情绪、成就、风险和 RAG 检索记录之间的关联关系。
11. 系统保存语音转文字结果，不保存用户原始语音。
12. 系统不保存生成图片文件，但可以保存图片 prompt 和模型调用元数据。
13. 研究数据默认以实名方式存储和导出，但导出权限仅开放给管理员。
14. 生产环境自建 MongoDB 必须开启鉴权，禁止无鉴权运行。
15. 生产环境 MongoDB 禁止直接暴露公网端口，应通过阿里云安全组、白名单、VPC 或等效网络隔离方式限制访问。
16. 应用后端、数据库管理、备份恢复等账号必须分离，业务应用不得使用 root 账号直连数据库。
17. 数据库连接密码和外部服务密钥不得写入代码仓库，必须通过环境变量或密钥管理方式注入。
18. 管理员使用 MongoDB Compass 等工具访问数据库时，必须受 IP 白名单或堡垒机/VPN 控制。

### 10.2 AI 安全

1. AI 不得进行医学诊断。
2. AI 不得鼓励或指导自伤、自杀或他伤行为。
3. AI 不得生成刺激性、恐怖、血腥、自伤相关图片。
4. 高风险时停止教学并触发风险提示。
5. 风险识别不能只依赖关键词，也应使用 AI 语义识别。
6. DBT 技能教学和测试内容应优先基于 RAG 知识库。

### 10.3 可用性

1. 面向中学生，文案应清晰、简短、自然。
2. 教学流程应自动推进。
3. 测试题应以具体情景为核心。
4. 每题解析应控制长度，避免过度复杂。
5. 页面操作应简单，避免过多选择。
6. 单次训练建议控制在 15-20 分钟内，以适配每天 1 次、持续 2-3 周的研究节奏。
7. 前端首版必须适配电脑与平板设备，不以手机为首版支持范围。
8. 教学页、测试页、情绪弹窗、风险弹窗、录音按钮和播放按钮必须针对触屏平板优化点击面积与布局。
9. 首版至少验证 Windows/Chrome 与 iPad/Safari 两类典型环境。

### 10.4 部署与访问环境

1. 预发布和生产环境必须使用 HTTPS，以满足浏览器麦克风权限要求。
2. 本地开发环境可使用 `localhost` 调试语音能力，但非本地访问场景不得依赖明文 HTTP。
3. 生产环境推荐使用正式域名或学校/项目已有子域名，不建议直接通过裸 IP 提供语音功能。
4. 扬声器播放虽然通常不需要单独授权，但仍需遵守浏览器自动播放策略并提供明确的用户交互入口。
5. 正式公网访问入口固定为 `https://<domain>:10443`，不使用 `80`、`443`、`8080` 作为正式业务端口。

### 10.5 跨设备访问与环境一致性

1. 系统必须支持“本机开发正常”与“其他电脑 / 平板访问正常”两种场景的一致性，不允许依赖仅本机可访问的地址配置。
2. 浏览器端请求不得写死 `localhost`、`127.0.0.1` 或本机临时 IP；前端与页面内脚本应优先使用相对路径访问同源接口。
3. 正式环境中，用户、学校老师和研究者应统一通过 `https://<domain>:10443` 访问系统，不应绕过 Nginx 直接访问 Django 服务端口。
4. Django、MongoDB、Redis、Qdrant、MinIO 等内部服务地址与对外访问地址必须分离配置，不得混用。
5. 登录态、Cookie、CSRF、语音权限、音频播放和可视化报告等能力，必须在跨设备场景下验证，而不能仅以本机浏览器结果作为通过标准。
6. 首版必须至少完成以下异机验证：
   - 本机浏览器
   - 同网络下另一台电脑浏览器
   - iPad / Safari
7. 若跨设备场景出现登录失败、会话丢失、录音失败、报告页异常、资源地址错误等问题，应优先排查同源策略、Cookie 设置、CSRF 配置、监听地址、反向代理配置与写死地址问题。

## 11. MVP 范围

### 11.1 MVP 必须包含

1. Web 用户邀请码注册与登录。
2. 注册后问卷，并允许用户后续修改。
3. 当前正念完整技能集合的教学支持。
4. 整理和补充现有 DBT 技能文档，并导入 RAG 知识库。
5. RAG 知识库，用于 DBT 教学与测试文档存储和查询。
6. MiniMax LLM API 作为优先候选，用于 AI 教学、测试生成、解析和风险语义判断。
7. 教学前情绪记录弹窗。
8. AI 信息收集。
9. AI 个性化技能选择。
10. 对话式教学。
11. 教学情景图片生成，默认建议使用阿里云通义万相。
12. 生成图片仅临时展示，不保存图片文件。
13. 教学完成后自动跳转测试。
14. 5 道情景选择题。
15. 测试题图片生成。
16. 每题即时解析。
17. 4/5 通过规则。
18. 失败讲解与重测。
19. 教学后情绪记录弹窗。
20. 中文成就系统。
21. 高风险识别：AI + 关键词。
22. 高风险自动停止教学。
23. 风险提示弹窗：热线 + 联系学校老师。
24. 文字输入、语音输入和 AI 语音播报。
25. MiniMax Speech 系列作为 TTS 优先候选。
26. 保存语音转文字结果，不保存原始语音。
27. 不支持实时语音通话，仅支持录音/转写/回复/播报。
28. 研究后台查看用户信息、教学记录、测试记录、情绪记录、成就记录、风险记录和 RAG 文档状态。
29. 报告查看用户前台只读查看可视化报告并导出 PDF 报告。
30. 权限分为学生、报告查看用户与管理员。
31. 电脑与平板双端适配。
32. 生产环境 HTTPS 部署与麦克风权限支持。
33. 管理员可导出原始数据，默认 JSON，可选 CSV / Excel；并可查看或导出可视化报告（PDF）。
34. 不需要匿名化导出。
35. 后台需要操作日志。

### 11.2 MVP 实施策略

虽然 MVP 范围保持完整，但实施上应采用分步推进策略。只有当前一步完成并验证无误后，才进入下一步，避免在研究场景中同时引入过多未验证能力。

建议顺序如下：

1. 账号、邀请码、问卷、基础后台与数据结构。
2. RAG 文档整理、上传、检索与教学链路。
3. 自动测试、逐题解析、重测与结果保存。
4. 情绪记录、成就系统与研究数据导出。
5. 风险识别与危机处理链路。
6. 图像生成与语音输入/播报能力。

### 11.3 可后续扩展

1. 扩展至 DBT 四大模块。
2. 更完整的 DBT 技能库。
3. 更丰富的情绪趋势图。
4. 更复杂的后台统计分析。
5. 数据匿名化导出配置。
6. 更细粒度的多角色权限与授权范围配置。
7. 更细粒度的 RAG 文档版本控制和内容审核流程。

## 12. 仍需确认的问题

为进入 PRD v1.0 和开发设计阶段，还需确认以下内容：

1. MiniMax 是否作为最终 LLM API 供应商。
2. MiniMax 是否作为最终 TTS 供应商。
3. 语音转文字是否最终采用 MiniMax；如 MiniMax ASR 不满足项目要求，需要选择中国大陆可用的替代 ASR 服务。
4. 是否需要找回密码。
5. 是否需要邮箱或手机号。
6. 是否允许用户注销账号。
7. 5 个 emoji 的具体样式。
8. 是否允许用户添加情绪备注。
9. 情绪历史页面采用列表、图表还是日历视图。
10. 现有 DBT 文档的格式、数量、来源和版权/使用授权状态。
11. 生产环境最终使用的正式域名或子域名。

## 13. 当前版本结论

当前版本中，平台的核心闭环为：

**邀请码注册登录 → 问卷建档 → 教学前情绪记录 → AI 信息收集 → RAG 检索 DBT 技能文档 → 个性化 DBT 正念技能教学 → 情景图片辅助 → 自动进入测试 → RAG 辅助生成 5 道图片情景选择题 → 逐题解析 → 失败讲解与无限重测或自主终止 → 教学后情绪记录 → 成就更新 → 后台记录与导出。**

平台的关键设计重点是：

1. 教学与测试自然衔接，避免用户手动选择。
2. 研究目标是验证该教学链路是否能够帮助已筛入受试者改善情绪。
3. 受试者由学校与精神卫生中心完成评估和筛入，平台不承担筛查职责。
4. 测试题由 AI 结合用户信息、历史记录和 RAG 文档自主生成，且技能顺序不固定。
5. 单次训练建议每天 1 次、每次 15-20 分钟，持续 2-3 周。
6. 成就系统保留，但取消经验与等级系统。
7. 高风险表达由 AI + 关键词识别，触发后停止当前会话，但不禁止后续再次进入。
8. 研究场景默认有实验人员陪同，因此系统无需额外通知研究人员。
9. AI 需支持 RAG，用于 DBT 技能教学与测试的文档存储、检索和生成依据。
10. 图像生成建议默认使用中国大陆可用的阿里云通义万相 API；腾讯混元生图和火山引擎即梦 AI 可作为备选。
11. 系统支持语音输入和 AI 语音播报，但只保存语音转写文本，不保存原始语音；当前不需要实时语音通话。
12. 生成图片不持久化保存，仅保存 prompt 和模型调用元数据。
13. 研究后台允许管理员查看并导出实名全链路研究数据，默认 JSON，可选其他格式。
14. 学校老师和其他研究者只能通过前台只读可视化报告查看学生情况，并可导出个体化报告，不可修改或删除原始数据。
15. 权限分为学生、报告查看用户与管理员，管理员操作需要记录日志。
16. 自建 MongoDB 是既定方案，必须纳入鉴权、账号分离、网络隔离和敏感访问控制。
17. 当前已有 DBT 技能文档，但需要整理、补充、切分、标注元数据并导入 RAG 知识库。
18. 前端首版需适配电脑与平板，重点验证 Windows/Chrome 与 iPad/Safari。
19. 生产环境需要 HTTPS 才能稳定支持麦克风权限，建议使用正式域名或子域名部署。
20. MVP 范围保持完整，但实施上按阶段推进，上一阶段验证无误后再进入下一阶段。
21. MiniMax 可作为 LLM API 和 TTS 的优先候选；语音转文字需要进一步确认其 ASR 能力是否满足项目要求。
