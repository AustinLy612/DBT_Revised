# DBT 技能教学平台实施计划

本计划基于 [DBT_PRD_v0.6.md](./DBT_PRD_v0.6.md) 和 [tech-stack.md](./tech-stack.md) 制定，目标是为 AI 开发者提供一组**可逐步执行、可验证、不可跳步**的实施指令。  
本计划只包含清晰具体的任务说明与验证要求，不包含代码。

---

## 0. 执行契约

### 0.1 文档定位

本文件是**开发执行文档（execution plan）**。  
它回答“按什么顺序做、每一步要产出什么、怎样才算通过，何时才能进入下一步”。

### 0.2 输入文档优先级

执行本文件前，默认按以下顺序读取输入：

1. `DBT_PRD_v0.6.md`：先确认做什么、不能做什么。
2. `tech-stack.md`：再确认用什么技术、通过什么部署与环境约束实现。
3. `inplementation-plan.md`：最后按步骤落地。

如果实施时发现本文件与前两者冲突，应暂停当前步骤并先回改本文件，而不是忽略冲突继续编码。

### 0.3 LLM 执行规则

模型在执行本文件时应遵循：

1. 每一步都必须同时完成功能、数据结构、界面入口和验证，不能只完成其中一部分。
2. 未通过本步验证前，不得进入下一步。
3. “完成标准”用于判断是否可进入下一阶段，“必须执行的验证测试”用于判断是否真正通过。
4. 本文件默认不要求一次性写完所有代码，而要求**按阶段交付可验证增量**。
5. 若某一步被待确认项阻塞，应显式提出问题，而不是私自补充需求或替换技术路线。

### 0.4 本文件的输出格式

每一步结束时，执行者都应能回答以下问题：

1. 本步新增了哪些模型、字段、页面、接口或后台入口？
2. 本步做了哪些配置或基础设施改动？
3. 本步执行了哪些测试？
4. 哪些风险已关闭，哪些风险仍然存在？
5. 是否允许进入下一步？

### 0.5 执行总原则

1. 必须严格按步骤推进，上一阶段未通过验证前，不得进入下一阶段。
2. 每一步都必须同时完成：
   - 功能实现
   - 数据结构落地
   - 最小可用界面或后台入口
   - 自动化测试或可重复人工测试
3. 每一步都必须记录：
   - 本步新增的数据模型或字段
   - 本步新增的接口、页面或后台入口
   - 本步验证结果
   - 未解决问题与风险
4. 严禁在核心流程未验证通过前提前接入复杂 AI 能力。
5. 教学与测试子流程使用 **LangChain-based RAG + structured output**，但注册、权限、风险中止、数据写入、导出等产品主流程必须由应用层显式控制。
6. 所有外部能力调用都必须有失败处理与日志记录，包括：
   - MiniMax LLM
   - MiniMax TTS
   - MiniMax 图像生成
   - MiniMax ASR 或其回退方案
7. 任何涉及高风险文本的流程，都必须优先满足“立即停止当前会话并记录风险事件”的产品规则。
8. 任何面向浏览器的功能都不得只以 `localhost` 场景作为通过标准，必须至少验证一台其他电脑或平板的访问结果。
9. 浏览器侧代码、模板和脚本不得写死 `localhost`、`127.0.0.1` 或开发机私有 IP 作为正式请求地址。

## 1. Step 1：初始化项目骨架与基础环境

### 指令

1. 初始化 Django 6.0 单体应用工程。
2. 创建一个统一的 Conda 主环境，用于承载本项目全部 Python 依赖。
3. 固定依赖管理方式，确保后续开发者可以复现同一环境。
4. 配置阿里云自建 MongoDB 连接与本地开发环境配置。
5. 接入 `django-mongodb-backend`。
6. 建立基础应用模块目录，至少包括：
   - 账号与邀请码
   - 问卷
   - 教学会话
   - 测试
   - 情绪记录
   - 风险控制
   - 知识库
   - 后台导出
   - 前台报告查看
7. 完成 MongoDB 最小安全基线配置，至少包括：
   - 开启鉴权
   - 创建 root / 应用 / 备份 等分离账号
   - 禁止应用使用 root 账号
   - 限制数据库监听地址和访问来源
8. 配置基础静态资源、模板目录、环境变量读取、日志输出规范。
9. 建立 Celery、Redis、MinIO、Qdrant 的运行配置，但本步不要求接入真实 AI 调用。
10. 配置 Nginx 反向代理骨架，明确正式业务入口监听 `10443`。
11. 设计域名与证书接入方案，明确使用 Let's Encrypt `DNS-01` 方式签发证书，不依赖 `80/443` 验证。
12. 建立环境配置约定，至少区分：
   - 内部服务地址
   - 对外访问基地址
   - Cookie / CSRF / Host 校验相关配置

### 完成标准

1. 项目可在本地启动。
2. Django 管理后台可登录。
3. MongoDB 连接稳定。
4. Celery worker 与 beat 能正常启动。
5. MinIO 可访问。
6. Qdrant 可访问。
7. 新开发者只需激活同一个 Conda 环境即可启动 Python 侧服务。

### 必须执行的验证测试

1. 启动验证：
   - Web 服务能启动并返回首页或占位页。
   - 后台登录页可访问。
   - Celery worker 无启动异常。
2. 数据连接验证：
   - MongoDB 可写入并读回一条测试记录。
   - MinIO 可成功上传并列出一个测试文件。
   - Qdrant 可创建并查询一个测试 collection。
3. 安全基线验证：
   - MongoDB 未开启鉴权时，部署检查应失败。
   - 应用账号可以访问业务库，但不能执行管理员级数据库操作。
   - root 账号不写入应用配置文件。
   - MongoDB 端口不对公网开放，或至少被阿里云安全组限制到预期来源。
4. 配置回归验证：
   - 删除或修改关键环境变量后，系统能给出明确错误提示，而不是静默失败。
5. 环境复现验证：
   - 在全新终端中仅通过激活同一个 Conda 环境，就能启动 Web 服务与后台任务。
   - 不依赖额外手工安装未记录的 Python 包。
6. 网关验证：
   - Nginx 配置中正式业务监听端口为 `10443`。
   - 系统未将 `80/443/8080` 作为正式业务入口。
7. 跨设备基础验证：
   - 通过域名入口从另一台设备访问占位页成功。
   - 项目代码中未出现用于正式请求的写死 `localhost` 地址。

## 2. Step 2：实现账号、邀请码与多角色基础权限

### 指令

1. 实现普通用户注册、登录、退出登录。
2. 注册流程强制要求输入有效邀请码。
3. 实现三类角色：
   - 学生用户
   - 报告查看用户
   - 管理员
4. 普通学生只能访问自己的数据。
5. 管理员可进入研究后台。
6. 报告查看用户只能进入前台只读报告系统。
7. 为邀请码建立最小管理能力，包括创建、启用、停用、查看使用情况。
8. 为报告查看用户建立最小授权能力，至少支持：
   - 创建查看账号
   - 启用 / 停用
   - 配置可查看学生范围

### 完成标准

1. 用户只能通过邀请码完成注册。
2. 无效邀请码无法注册。
3. 三类角色访问范围明确隔离。
4. 后台可查看邀请码使用情况。
5. 报告查看用户不能进入后台管理页面。

### 必须执行的验证测试

1. 正向测试：
   - 使用有效邀请码成功注册。
   - 注册后可登录、退出、再次登录。
2. 负向测试：
   - 无邀请码时注册失败。
   - 失效邀请码注册失败。
   - 普通用户访问后台页面时被拒绝。
3. 权限测试：
   - 用户 A 无法查看用户 B 的任何记录。
   - 管理员可查看用户列表。
   - 报告查看用户只能看到被授权学生的报告入口。
   - 报告查看用户无法修改或删除原始数据。
4. 会话一致性测试：
   - 通过域名入口在另一台设备上登录后可保持会话。
   - 不因 Host、Cookie、CSRF 配置错误导致异机登录失败。

## 3. Step 3：实现核心数据模型与后台最小可视化

### 指令

1. 按 PRD 建立核心数据模型，至少覆盖：
   - User
   - UserProfile
   - TeachingSession
   - ChatMessage
   - Test
   - TestQuestion
   - MoodRecord
   - Achievement
   - UserAchievement
   - RiskEvent
   - KnowledgeDocument
   - KnowledgeChunk
   - RetrievalLog
   - AdminOperationLog
   - ReportAccessLog
2. 为每个模型建立后台可查看入口。
3. 为关键模型建立最小列表页与详情页，优先级最高的是：
   - 用户
   - 教学会话
   - 测试
   - 风险事件
   - RAG 文档
   - 报告访问日志
4. 明确各模型之间的关联关系和状态字段。

### 完成标准

1. 所有 PRD 中的核心对象均可在后台查看。
2. 关键状态字段可读且语义一致。
3. 后台可以基于用户聚合查看主要记录。
4. 用户角色字段与报告访问日志结构可支持三类权限模型。

### 必须执行的验证测试

1. 模型一致性测试：
   - 每个模型至少能创建、读取、更新一条示例数据。
2. 后台可用性测试：
   - 管理员可以从后台进入关键模型列表和详情页。
3. 关联验证：
   - 一个用户的教学、测试、情绪、风险记录能在后台被关联查看。
   - 一个报告查看用户的查看 / 导出操作能在日志中被追踪。

## 4. Step 4：实现注册后问卷与问卷修改流程

### 指令

1. 实现注册后强制进入问卷页面。
2. 问卷必须覆盖 PRD 已确认字段：
   - 性别
   - 年龄
   - 年级
   - 爱好标签
   - 苦恼/困扰标签
   - 其他补充文本
3. 用户首次未完成问卷前，不允许进入教学主页。
4. 用户后续允许修改问卷。
5. 问卷更新后，后续教学与测试必须使用最新问卷信息。

### 完成标准

1. 新用户注册后必须完成问卷。
2. 问卷完成状态可被系统识别。
3. 问卷修改后能立即生效。

### 必须执行的验证测试

1. 流程测试：
   - 新注册用户自动跳转问卷页。
   - 未完成问卷时无法开始教学。
2. 数据测试：
   - 提交问卷后后台可查看完整问卷记录。
   - 修改问卷后更新时间正确变化。
3. 规则测试：
   - 爱好与困扰标签支持多选。
   - “其他”补充文本可保存。

## 5. Step 5：实现知识库上传、切分、索引与检索日志

### 指令

1. 实现后台上传 DBT 原始文档。
2. 文档上传后触发异步处理：
   - 文档解析
   - 文本切分
   - metadata 生成
   - embedding 生成
   - MongoDB text index / Qdrant 索引写入
3. metadata 至少包含：
   - document_id
   - version
   - module
   - skill
   - section_title
   - difficulty
   - is_beginner_friendly
   - scenario_tags
   - risk_flags
4. 建立文档状态流转，至少包括：
   - 已上传
   - 处理中
   - 可检索
   - 失败
5. 实现检索日志写入机制。

### 完成标准

1. 后台上传文档后可看到处理状态。
2. 文档处理成功后可检索到相关 chunk。
3. 每次检索都能生成 RetrievalLog。

### 必须执行的验证测试

1. 上传处理测试：
   - 上传一份合法文档后状态从“处理中”变为“可检索”。
   - 上传异常文档后状态变为“失败”，并保留错误信息。
2. 检索测试：
   - 用技能名检索能返回对应 chunk。
   - 用场景语义检索能返回合理片段。
3. 日志测试：
   - 每次检索后都生成 RetrievalLog，并包含查询内容、用途、返回 chunk 列表。

## 6. Step 6：实现 LangChain-based RAG 基础能力

### 指令

1. 用 LangChain 封装教学与测试子流程所需的 RAG 能力。
2. 明确建立以下可复用模块：
   - 检索器
   - 提示词模板
   - structured output schema
   - 输出校验器
3. 所有关键输出都必须结构化，至少包括：
   - 技能选择结果
   - 教学计划
   - 教学摘要
   - 测试题
   - 正确答案
   - 题目解析
   - 风险语义判定结果
4. 当检索结果不足时，模型调用自身知识进行回答与教学，但禁止编造具体 DBT 细节。
5. RAG 检索器必须支持：
   - MongoDB 关键词检索
   - Qdrant 语义检索
   - 应用层结果合并与去重

### 完成标准

1. LangChain 子流程可独立接收输入并输出结构化结果。
2. 输出格式稳定，不依赖脆弱的字符串解析。
3. 检索结果不足时，系统行为可预测。

### 必须执行的验证测试

1. 结构化输出测试：
   - 每类输出都能通过 schema 校验。
2. 检索依赖测试：
   - 在提供有效检索上下文时，输出字段完整。
3. 稳定性测试：
   - 对同一输入重复调用多次，输出结构保持一致。

## 7. Step 7：实现教学会话主流程

### 指令

1. 实现单次教学会话的状态机。
2. 至少覆盖以下状态：
   - pre_mood_recording
   - info_collection
   - skill_selection
   - rag_retrieval_for_teaching
   - teaching
   - completed
   - stopped_by_risk
   - user_terminated
3. AI 教学流程必须能够：
   - 读取用户问卷
   - 读取历史教学和测试记录
   - 检索 RAG 内容
   - 生成个性化技能选择
   - 生成教学内容
   - 保存教学摘要
4. 会话中的聊天消息必须逐条保存。

### 完成标准

1. 用户可以完成一次完整教学会话。
2. 教学会话状态变化可在后台查看。
3. 聊天消息和教学摘要可追溯。

### 必须执行的验证测试

1. 流程测试：
   - 从开始到教学结束可完整跑通。
2. 状态测试：
   - 每一步进入和退出时状态正确变化。
3. 数据测试：
   - 会话结束后可查看：
     - 选择的技能
     - 选择理由
     - 使用的 RAG 片段
     - 教学摘要
     - 聊天记录

## 8. Step 8：实现测试、逐题解析与无限重测

### 指令

1. 在教学完成后自动进入测试。
2. 测试必须由 LangChain 子流程基于：
   - 用户信息
   - 教学内容
   - 历史记录
   - RAG 检索结果
   生成 5 道情景选择题。
3. 每题必须同时生成：
   - 题干
   - 选项
   - 正确答案
   - 解析
   - 支撑用的 source chunk
4. 用户每答一题后立即得到解析。
5. 正确 4 题及以上判定通过。
6. 未通过后允许无限重测。
7. 用户允许主动终止当前测试。

### 完成标准

1. 教学后自动进入测试。
2. 测试记录、题目、答案、解析全部可追踪。
3. 重测不重复原题。

### 必须执行的验证测试

1. 正向测试：
   - 5 题测试完整跑通。
   - 4/5 与 5/5 被正确判定为通过。
2. 负向测试：
   - 3/5 及以下被判定为未通过。
3. 逐题解析测试：
   - 每答一题后立即展示解析。
4. 重测测试：
   - 连续两次未通过后，系统仍允许继续重测。
   - 重测题目与原题不完全重复。
5. 中止测试：
   - 用户主动终止后，测试状态记录为 `user_terminated`。

## 9. Step 9：实现情绪记录与成就系统

### 指令

1. 实现教学前情绪记录弹窗。
2. 实现教学后情绪记录弹窗。
3. 支持手动情绪记录与历史查看。
4. 情绪记录使用 5 个 emoji 方案。
5. 成就系统至少实现 PRD 中当前列出的成就。
6. 成就触发必须依赖真实业务记录，不允许仅前端展示。
7. 为后续个体可视化报告预留统计聚合接口。

### 完成标准

1. 每次教学前后都能记录情绪。
2. 情绪历史可查看。
3. 成就会在满足条件时自动解锁。

### 必须执行的验证测试

1. 情绪流程测试：
   - 教学开始前弹出情绪记录。
   - 测试结束后弹出情绪记录。
2. 数据测试：
   - 情绪记录可在后台与用户和会话关联查看。
3. 成就测试：
   - 完成首次训练后触发“第一步”。
   - 5/5 正确时触发“满分通过”。
   - 首次情绪记录时触发“情绪记录开始”。

## 10. Step 10：实现风险识别与高风险中止

### 指令

1. 实现双通道风险识别：
   - 关键词识别
   - LLM 语义识别
2. 在教学和测试两个阶段都接入风险识别。
3. 一旦判定高风险，必须立即：
   - 停止当前会话
   - 保存上下文
   - 记录 RiskEvent
   - 弹出风险提示
4. 当前会话中止后，不继续普通教学。
5. 高风险中止只影响当前会话，不封禁后续登录。

### 完成标准

1. 高风险输入会触发中止。
2. 风险事件可在后台查看。
3. 中止后的会话状态正确。

### 必须执行的验证测试

1. 关键词风险测试：
   - 输入明确高风险词句后，会话立即中止。
2. 语义风险测试：
   - 输入不含关键词但语义高风险的句子后，会话中止。
3. 误报控制测试：
   - 正常情绪困扰表达不应被大面积误判为高风险。
4. 数据测试：
   - RiskEvent 包含触发文本、来源、动作、follow_up_mode。
5. 会话恢复测试：
   - 高风险中止后，用户下次仍可重新进入系统开始新会话。

## 11. Step 11：实现 MiniMax 图像、TTS 与 ASR 集成

### 指令

1. 接入 MiniMax 图像生成：
   - 默认 `image-01`
   - 低延迟可选 `image-01-live`
2. 接入 MiniMax TTS：
   - 默认 `speech-2.8-turbo`
   - 高质量可选 `speech-2.8-hd`
3. 接入 ASR：
   - 优先 MiniMax
   - 若当前账号不可用则按技术栈切到火山引擎回退方案
4. 图像生成用于：
   - 教学情景
   - 测试题配图
5. 语音用于：
   - 用户录音输入
   - AI 语音播报
6. 必须符合 PRD 的数据约束：
   - 不保存原始音频
   - 不持久化保存生成图片文件
   - 可保存 prompt、模型名、临时链接、时间戳等元数据
7. 预发布与生产环境的语音链路测试必须在 HTTPS 环境下执行。
8. 预发布与生产环境统一以 `https://domain:10443` 形式访问。

### 完成标准

1. 教学与测试均可生成图片。
2. AI 回复可播报。
3. 用户可通过语音输入。
4. 失败时有清晰回退与错误提示。

### 必须执行的验证测试

1. 图像生成测试：
   - 教学页生成一张场景图。
   - 测试页生成一张题目配图。
   - 后台可查看图像生成元数据，但不保存图片文件。
2. TTS 测试：
   - 一条 AI 回复可成功播报。
   - 用户关闭播报开关后不再自动播放。
3. ASR 测试：
   - 用户上传录音后得到转写文本。
   - 系统只保存转写，不保存原始音频。
4. 异常测试：
   - 外部服务超时后，页面有明确失败提示，且主会话不因一次失败而崩溃。
5. 浏览器权限测试：
   - 在 HTTPS 环境中可正常申请麦克风权限。
   - 在不满足安全上下文时，页面给出明确提示而非静默失败。
6. 端口测试：
   - 在 `https://domain:10443` 下可正常完成录音与播报。
   - 未要求 `80/443/8080` 提供正式业务访问。
7. 资源地址测试：
   - 语音、图片、报告等前端可见资源地址不指向 `localhost`、`127.0.0.1` 或开发机私有 IP。

## 13. Step 13：非功能验证与上线前验收

### 指令

1. 对全系统进行一次端到端回归验证。
2. 对核心链路进行失败场景测试，包括：
   - LLM 失败
   - 检索失败
   - 图片生成失败
   - TTS 失败
   - ASR 失败
   - MongoDB 短暂不可用
3. 检查日志、错误提示、数据一致性和后台审计。
4. 对所有 PRD 的 P0 项进行逐条对照验收。
5. 对电脑与平板双端进行回归测试。
6. 对正式公网入口 `10443` 进行访问回归测试。

### 完成标准

1. 所有 P0 功能可用。
2. 关键失败场景不会导致数据损坏或流程卡死。
3. 后台可以追溯主要研究过程。
4. 电脑与平板关键页面可正常使用。
5. `https://domain:10443` 可作为稳定正式入口使用。

### 必须执行的验证测试

1. 端到端主链路测试：
   - 邀请码注册
   - 问卷完成
   - 教学前情绪记录
   - 教学
   - 自动测试
   - 逐题解析
   - 教学后情绪记录
   - 后台查看与导出
2. 设备兼容性测试：
   - Windows / Chrome 下完成完整主链路
   - iPad / Safari 或 Playwright WebKit 等效环境下完成教学、测试、录音、播报和报告查看链路
3. 风险链路测试：
   - 在教学中触发高风险后被立即中止。
4. 数据追踪测试：
   - 同一用户在后台可看到完整链路。
5. 回归测试：
   - 所有前序步骤中的关键测试重新执行一次并通过。
6. 正式入口测试：
   - 通过 `https://domain:10443` 完成完整访问。
   - Let's Encrypt `DNS-01` 证书链有效，浏览器不报不受信任错误。
7. 环境一致性排查测试：
   - 浏览器开发者工具中不存在因跨域导致的主链路失败。
   - 页面请求、音频地址、报告资源地址未指向 `localhost`、`127.0.0.1` 或开发机私有 IP。
   - 通过域名入口访问时，Cookie、CSRF、重定向与登录态在异机上行为一致。
